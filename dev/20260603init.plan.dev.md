# vaultpub 模块化开发目标

本文档是 `dev/20260603init.plan.md` 的执行版开发规格，用于直接指导后续编码。原参考文档中的能力不删减，但本文件按模块、依赖关系、阶段验收和开发记录要求重组，避免开发时一次性铺开过多功能。

项目统一命名为 `vaultpub`：

- Python distribution name: `vaultpub`
- Python import package: `vaultpub`
- CLI 命令: `vaultpub`
- Django app namespace: `vaultpub.django_app`
- 默认配置文件: `.vaultpub.yml`
- 可选兼容旧草案配置文件名: `.obsidian-publish.yml`

原文中的 `obsidian-publish-py`、`obsidian_publish` 只作为历史草案名处理，后续实现、测试、文档不得继续使用这些名称，除非是在迁移说明中明确说明。

## 1. 产品目标

`vaultpub` 是一个 Python package，用于把本地 Obsidian vault 发布为可浏览、可搜索、可集成的 Web 站点。它不是 Obsidian 官方 Publish 的客户端，不依赖 Obsidian 账号、订阅、官方服务器或私有 API。

核心目标：

1. 给定本地 vault 路径，扫描 Markdown、附件、frontmatter、链接、标签、标题和块引用。
2. 按发布规则过滤 private、excluded、hidden 和 `publish: false` 内容。
3. 渲染 Obsidian 风格 Markdown，包括 wikilinks、embeds、callouts、tags、Mermaid、Math、backlinks、TOC。
4. 生成文件导航、搜索索引、链接关系图谱、反链、标签页和 SEO 元数据。
5. 提供独立 ASGI Web 服务：`vaultpub serve --vault /path/to/vault`。
6. 提供静态导出：`vaultpub build --vault /path/to/vault --out ./public`。
7. 提供 Django reusable app：可加入 `INSTALLED_APPS` 并通过 `include("vaultpub.django_app.urls")` 集成。
8. 支持实时更新：文件变更后增量索引，并通过 SSE/WebSocket/poll 更新前端。
9. 默认安全：不泄露 vault 绝对路径，不直接公开 `.md`、`.obsidian`、`.git`、`metadata.json`、hidden/private 内容。

首要判断标准：先让“一个 vault 能安全、正确、快速地被浏览”成立，再逐步补齐 Perlite 和 Obsidian Publish 的高级体验。

## 2. 非目标

1. 不实现 Obsidian 官方 Publish 的上传、同步、账号认证或 headless publish API。
2. 不读取或执行 Obsidian 插件代码。
3. 不在默认模式执行用户 vault 中的 JavaScript。
4. 不依赖数据库作为必需运行条件；SQLite FTS 只能作为可选增强。
5. 不把浏览器直接暴露给 vault 原始 Markdown 或 metadata 文件。
6. v0.1 不追求完整 Obsidian 渲染兼容；v0.1 目标是安全可用的基础浏览、链接和集成。

## 3. 顶层架构

采用“框架无关 core + 多入口适配”的架构。

```text
+--------------------+        +---------------------+
| vaultpub.cli       | -----> | vaultpub.core       |
| vaultpub.web       | -----> | config / scan       |
| vaultpub.django_app| -----> | index / render      |
| vaultpub.export    | -----> | safety / events     |
+--------------------+        +---------------------+
             |
             v
       frontend assets
```

### 3.1 core

`vaultpub.core` 是唯一业务内核，必须保持 Web 框架无关。

职责：

- 配置加载与归一化。
- vault 路径安全、相对路径规范、URL path 生成。
- 文件扫描、发布过滤、首页解析、附件识别。
- frontmatter/properties 解析。
- Obsidian Markdown 语法解析、结构元数据提取。
- 链接解析、歧义处理、broken link 记录。
- `VaultIndex`、nav tree、tags、backlinks、graph、search documents。
- HTML 渲染、sanitize、SEO 元数据、cache key。
- watcher 事件、增量索引、事件广播的框架无关模型。

硬性约束：

- `core` 不得 import Django、Starlette、FastAPI、Channels。
- 所有路径操作必须走 `core.paths`。
- public function 必须有类型注解。
- 用户笔记内容不得进入 `eval` / `exec`。

### 3.2 web

`vaultpub.web` 是独立 ASGI 入口。

职责：

- `create_app(config: PublisherConfig)`。
- 页面路由、附件路由、API 路由、SSE/WebSocket 路由。
- Jinja 模板渲染。
- 静态资源服务或挂载。
- 调用 core，不复制 core 逻辑。

### 3.3 django_app

`vaultpub.django_app` 是 Django reusable app 适配层。

职责：

- `AppConfig`。
- `VAULTPUB` settings 解析。
- `urls.py`、views、templates、static。
- optional Channels consumer。
- 在未安装 Channels 时提供 SSE 或 poll fallback。
- 多站点配置时通过 site id 获取对应 publisher。

约束：

- Django 层不重新实现扫描、解析、索引、渲染算法。
- Django settings 中的路径、prefix、feature flags 必须归一化为 `PublisherConfig`。

### 3.4 cli

`vaultpub.cli` 提供命令行入口。

必须支持：

```bash
vaultpub --help
vaultpub serve --vault ~/Vault --home README --port 8008
vaultpub build --vault ~/Vault --out ./public
vaultpub index --vault ~/Vault --json ./index.json
vaultpub doctor --vault ~/Vault
vaultpub init --vault ~/Vault
```

CLI 使用 Typer。命令实现只负责参数解析、输出报告、调用 core/web/export。

### 3.5 export

`vaultpub.export` 使用 core 的同一套索引和渲染结果生成静态站点。

职责：

- 渲染所有 published notes。
- 输出附件、assets、tag pages、search-index、graph、sitemap、rss、robots。
- 支持 `base_url`、`clean`、pretty URLs。
- 静态模式不能依赖服务端 API 才能完成基础浏览。

### 3.6 frontend

`frontend` 用 Vite + TypeScript 构建，产物进入 package static。

职责：

- theme toggle。
- search modal。
- hover preview。
- graph rendering。
- Mermaid 和 KaTeX 初始化。
- callout folding。
- realtime client。
- mobile drawer / responsive controls。

原则：

- 服务端输出稳定 HTML 和数据端点。
- 前端只增强交互，不承担发布过滤和安全判定。

## 7. Data Model

### 7.1 Path And ID Rules

内部统一使用 POSIX 相对路径：

- `Folder/My Note.md`
- `Attachments/image.png`

公开 URL 默认不包含 `.md` 后缀：

- `Folder/My Note.md` -> `/Folder/My%20Note`
- `README.md` -> `/README`
- root `/` -> resolved home note

禁止把绝对路径作为公开 ID。Note ID 使用相对路径生成稳定 hash，大小写保留。

首页解析顺序：

1. `home_file` 配置。
2. `README.md`。
3. `index.md`。
4. `Home.md`。
5. 按导航排序后的第一个可见 Markdown 文件。

## 8. Security Model

### 8.1 Path Safety

必须实现 `safe_join`：

```python
def safe_join(vault_root: Path, rel: str) -> Path:
    candidate = (vault_root / rel).resolve()
    root = vault_root.resolve()
    if not candidate.is_relative_to(root):
        raise PathTraversalError(rel)
    return candidate
```

要求：

1. 禁止 `../` 逃逸。
2. 默认不跟随 symlink。
3. symlink 开启时也必须验证 resolved path 在允许 root 内。
4. 不暴露 vault 绝对路径。
5. 不允许直接访问 `.git`、`.obsidian`、隐藏文件、配置文件。
6. 不直接公开 `metadata.json`、原始 `.md`，除非后续明确配置。
7. 附件访问必须检查是否在 published set 或允许目录内。

### 8.2 Publish Filter

发布判断函数语义：

```text
if hidden and not hidden_file_access: hide
if matches excluded folder:
    publish: true may override only when allowed by mode
if matches excluded glob: hide
if publish_property_mode == publish_true:
    require publish: true
if publish: false: hide
otherwise publish
```

默认模式为 `publish_false_hides`。

### 8.3 HTML Safety

默认 `html_safe_mode = true`：

1. raw HTML 默认禁用或 sanitize。
2. SVG 默认作为 `<img>`，不 inline 未净化 SVG。
3. Mermaid 使用 strict security。
4. `publish.js` 默认只在 trusted mode 启用。
5. 外链加 `rel="noopener noreferrer"` 和 `target="_blank"`。
6. 文档必须提供 CSP 建议。

## 9. Markdown And Obsidian Syntax

### 9.1 Render Pipeline

每个 Markdown 文件的处理流程：

```text
read file
  -> decode UTF-8 / UTF-8 BOM
  -> split YAML frontmatter
  -> extract properties
  -> strip Obsidian comments for visible output
  -> parse structural metadata
  -> resolve links against VaultIndex
  -> transform Obsidian syntax into markdown-it tokens / placeholders
  -> markdown-it-py render
  -> postprocess anchors, embeds, callouts, Mermaid, Math
  -> sanitize HTML if safe_mode
  -> wrap page template
```

不要用正则直接渲染整个 Markdown。可以用正则识别 Obsidian 扩展语法，但完整 Markdown 渲染必须交给 parser。

### 9.2 Frontmatter

支持字段：

```yaml
---
title: Custom Title
aliases:
  - Old Name
tags:
  - project/demo
publish: true
permalink: custom/path
description: SEO description
image: assets/card.png
cover: assets/cover.png
cssclasses:
  - wide-page
---
```

规则：

1. `title` 优先作为页面标题。
2. 没有 `title` 时，使用第一个 H1。
3. 没有 H1 时，使用文件名 stem。
4. `aliases` 用于链接解析和旧 URL redirect。
5. `tags` 与正文 inline tags 合并。
6. `publish: false` 默认隐藏。
7. `publish: true` 在 `publish_true` 模式下发布，也可按规则覆盖 excluded folder。
8. `permalink` 覆盖默认 URL，默认路径重定向到 permalink。
9. `description`、`image`、`cover` 用于 SEO/OpenGraph。

### 9.3 Wikilinks

必须支持：

```md
[[Note]]
[[Note|显示文本]]
[[Folder/Note]]
[[Folder/Note.md]]
[[Note#Heading]]
[[Note#Heading|显示文本]]
[[Note#^block-id]]
[[#Current Page Heading]]
[[^current-page-block]]
![[Image.png]]
![[Image.png|300]]
![[Image.png|300x200]]
![[Other Note]]
![[Other Note#Heading]]
![[Other Note#^block-id]]
```

解析要求：

1. code fence、inline code 内不解析 wikilink。
2. Obsidian comment `%% ... %%` 中不解析链接。
3. `![[...]]` 为 embed。
4. `[[...]]` 为普通 link。
5. `#` 后为 heading/block anchor。
6. `#^` 表示 block reference。
7. `|` 对普通链接表示 display text。
8. `|100` 或 `|100x200` 对图片 embed 表示尺寸。
9. 解析失败时保留可见文本，并渲染 broken-link 样式。

### 9.4 Link Resolver

`LinkResolver.resolve(source_note, target_text)` 优先级：

1. 外部 URL 保持外链。
2. target 带扩展名且匹配附件：返回 attachment URL。
3. 相对路径先相对当前文件夹查找，再相对 vault root 查找。
4. 绝对 vault 路径相对 vault root 查找。
5. 不带 `.md` 时尝试 `target + ".md"`、stem、alias。
6. 多候选时优先同目录、最短路径、自然排序，并记录 warning。
7. 找不到时返回 unresolved link。

broken link HTML：

```html
<a class="internal-link is-unresolved" data-target="Missing">Missing</a>
```

### 9.5 Embeds

支持类型：

| 类型 | 渲染 |
| --- | --- |
| Markdown note | 递归渲染目标 note body |
| Heading | 渲染该 heading 到下一个同级/更高级 heading |
| Block | 渲染 block 内容 |
| Image | `<img>`，支持宽高 |
| SVG | 默认 `<img>`，不 inline |
| Audio | `<audio controls>` |
| Video | `<video controls>` |
| PDF | `<iframe>` 或 `<object>` |
| Canvas | 后续只读简化渲染 |
| Unsupported | 下载/打开链接 |

防递归：

```text
max_embed_depth = 5
if target_note in render_stack:
    render embed error: circular embed
```

### 9.6 Callouts

支持：

```md
> [!note] Title
> Content

> [!tip] Custom title
> [!warning]+ Expanded foldable
> [!faq]- Collapsed foldable
```

默认类型：

```text
note, abstract, info, todo, tip, success, question, warning,
failure, danger, bug, example, quote
```

alias：

```text
summary/tldr, hint/important, check/done, help/faq,
caution/attention, fail/missing, error, cite
```

未知类型降级为 `note`。

输出结构：

```html
<div class="callout" data-callout="tip" data-callout-fold="open">
  <div class="callout-title">
    <span class="callout-icon"></span>
    <span class="callout-title-inner">Custom title</span>
  </div>
  <div class="callout-content">...</div>
</div>
```

### 9.7 Tags

支持正文标签和 frontmatter 标签：

```md
#project/demo
#会议
#inbox/to-read
```

规则：

1. code block、inline code、URL fragment 内不识别 tag。
2. frontmatter `tags` 支持 list 和 comma string。
3. tag 比较时小写归一，展示时保留原始形式。
4. nested tags 建树。
5. tag 页面 URL：`/tags/project/demo`。

### 9.8 Backlinks

实现：

1. `linked_mentions`：其他 note 明确链接到当前 note。
2. `unlinked_mentions`：其他 note 正文中出现当前 note title / aliases，但没有链接；默认关闭。



## 11. Search

默认客户端搜索，构建 `search-index.json`：

```json
[
  {
    "id": "sha1...",
    "title": "README",
    "path": "README.md",
    "url": "/README",
    "content": "plain text",
    "tags": ["demo", "readme"],
    "headings": ["Intro", "Install"],
    "aliases": ["Home"],
    "excerpt": "A web based markdown viewer..."
  }
]
```

MiniSearch 字段：

- search fields: `title`, `aliases`, `tags`, `headings`, `content`
- store fields: `title`, `url`, `excerpt`, `tags`
- boost: title 4, aliases 3, tags 2, headings 2
- fuzzy: 0.2
- prefix: true

可选 SQLite FTS5：

```yaml
search:
  backend: sqlite
  sqlite_path: .cache/vaultpub/search.db
```

最小查询语法：

| 语法 | 含义 |
| --- | --- |
| `word` | 普通全文 |
| `"exact phrase"` | 短语 |
| `tag:#project` | tag |
| `path:folder` | 路径 |
| `title:foo` | 标题 |
| `-draft` | 排除词 |

## 12. Graph

数据结构：

```python
@dataclass
class GraphNode:
    id: str
    label: str
    title: str
    url: str | None
    group: Literal["note", "tag", "attachment"]
    value: int = 1

@dataclass
class GraphEdge:
    source: str
    target: str
    kind: Literal["link", "embed", "tag", "attachment"]
```

`GET /api/graph`：

```json

  "edges": [

}
```

本地图谱：

- `GET /api/graph/local/README?depth=2`
- 包含 outgoing links。
- 包含 backlinks。
- depth 可配置。
- 可选包含 tags。
- 可选隐藏 orphan nodes。

## 14. Realtime

使用 `watchfiles` 监听 vault：

```python
async def watch_vault(config: PublisherConfig, indexer: VaultIndexer, bus: EventBus):
    async for changes in awatch(config.vault_path):
        affected = classify_changes(changes)
        result = await indexer.apply_changes(affected)
        await bus.publish(IndexChangedEvent(result))
```

要求：

1. debounce 默认 150ms。
2. 忽略 `.swp`、`~`、`.tmp`、`.DS_Store`。
3. 忽略 `.obsidian/workspace*` 等高频变化。
4. 删除文件时更新 backlinks、graph、search、nav。
5. 重命名可按 delete + add 处理。
6. 文件变更后只重新解析受影响 note，再更新依赖该 note 的 backlinks/embeds。

事件格式：

```json
{
  "type": "index.changed",
  "version": 42,
  "changed": [
    {"kind": "note", "path": "README.md", "url": "/README", "change": "modified"}
  ],
  "deleted": [],
  "graph_changed": true,
  "nav_changed": false,
  "search_changed": true
}
```

前端行为：

1. 当前页面被修改：重新拉取 HTML partial 并替换 `.markdown-body`。
2. 当前页面被删除：显示 404，并推荐相近页面。
3. nav 变化：刷新左侧导航。
4. graph 变化：刷新 graph data。
5. search 变化：重新加载 search index。
6. backlinks 变化：刷新右侧 backlinks。
7. 不覆盖用户滚动位置，刷新后尽量恢复 heading anchor 和 scroll offset。

## 21. Development Record Requirement

每实现一个功能，都在 `dev/` 新建记录文件。建议命名：

```text
dev/YYYYMMDDNN-feature-name.md
```

模板：

```md
# Feature: <name>

## Goal

说明本次功能要解决什么问题，验收标准是什么。

## Conclusion

说明实际完成了什么，和目标相比是否有偏差。

## Changed Files

- ...

## Tests

- ...

## Manual Verification

- ...

## Known Limitations

- ...
```

注意：`dev/` 是项目治理文档目录，不应作为 scratch 忽略。若 `.gitignore` 忽略了 `dev/`，实施前应调整忽略规则，至少确保开发目标和功能记录能进入版本管理。

## 22. Implementation Order

推荐严格按此顺序开发：

1. 项目骨架与 `vaultpub --help`。
2. 安全路径与发布过滤。
3. vault 扫描和首页规则。
4. Markdown 基础渲染。
5. Wikilink 解析和 resolver。
6. 页面路由和 nav tree。
7. TOC、backlinks。
8. search documents 和 search API。
9. graph data 和 graph API。
10. Django 基础集成。
11. embeds、callouts、Mermaid、Math。
12. 前端布局与交互。
13. watcher 和 realtime。
14. static export。
15. SEO、RSS、sitemap、robots。
16. security hardening、doctor、benchmark、docs。
