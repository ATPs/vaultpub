# Python Obsidian Vault Publisher 开发指引

以下内容可以直接保存为 `DEVELOPMENT_GUIDE.md`，作为交给 Codex 的开发规格。项目目标是开发一个 Python package：给定一个 Obsidian vault 路径后，能够像 Perlite / Obsidian Publish 一样实时发布、渲染、搜索、展示链接关系，并且可以作为 Django reusable app 集成到现有 Django 项目中。

## 0. 目标与参考依据

### 0.1 产品目标

开发一个 Python 包，暂定名：

```text
obsidian-publish-py
import package: obsidian_publish
```

核心能力：

1. 指定本地 Obsidian vault 路径，自动构建可访问的网站。
2. 没有 URL 路径时，默认渲染首页文件；如果没有配置首页，则按规则渲染第一个可见 Markdown 文件。
3. 支持实时渲染：文件变更后自动重新索引、重新渲染、刷新前端视图。
4. 支持独立运行：`obsidian-publish serve --vault /path/to/vault`。
5. 支持静态导出：`obsidian-publish build --vault /path/to/vault --out ./public`。
6. 支持 Django 集成：作为 reusable app 引入 `INSTALLED_APPS` 和 `urls.py`。
7. 尽可能覆盖 Perlite 与 Obsidian Publish 的功能：导航、搜索、双链、悬浮预览、图谱、目录、主题、标签、附件、LaTeX、Mermaid、SEO、密码保护、publish.css / publish.js 等。

Perlite 的 README 将其定位为“web based markdown viewer optimized for Obsidian Notes”，核心思想是把整个 vault 或 Markdown 文件夹放入 Web 目录后页面自行构建，并列出了自动按文件夹结构构建、无数据库、Obsidian 主题、响应式、交互式图谱、LaTeX/Mermaid、Obsidian 链接、搜索、标签/链接/图片/预览、暗色/亮色模式等功能。([GitHub][1]) Obsidian Publish 官方强调 hover previews、graph view、stacked pages、backlinks，并提供主题、域名、密码保护、SEO、移动端、Analytics 等站点能力。([Obsidian][2])

### 0.2 非目标

本项目不是 Obsidian 官方 Publish 服务的客户端，也不依赖 Obsidian 账号、订阅、官方服务器或私有 API。官方 Obsidian Publish 现在也提供 headless client，用于 CI / 自动化发布；这个项目只借鉴其本地路径、首页、导航、图谱、搜索、反链、主题等配置模型，不调用官方发布接口。([Obsidian][3])

## 1. 总体架构

### 1.1 推荐架构

采用“三层 + 两种入口”的架构：

```text
obsidian_publish/
  core/          # 与 Web 框架无关的 vault 读取、索引、解析、渲染
  web/           # ASGI 独立站点：Starlette/FastAPI 风格
  django_app/    # Django reusable app 适配层
  cli/           # 命令行：serve, build, index, doctor
  frontend/      # 编译后的 JS/CSS 静态资源
```

核心原则：

1. `core` 不依赖 Django。
2. Django 集成层只负责 settings、URL、view、template、static、Channels/SSE 适配。
3. 独立运行模式使用 ASGI app，方便实时连接和静态部署预览。
4. 默认不需要数据库；可选使用 SQLite 缓存和全文搜索。
5. 所有功能都围绕一个 `VaultIndex` 内存索引工作。
6. 文件变更通过 watcher 触发增量索引，再广播给前端。

Django 官方文档建议将可复用 Django app 做成独立 Python package，并将 app 自身的 templates/static/tests/urls/views 放在 app 包内，方便插入其他项目。([Django Project][4]) Python 打包侧使用现代 `pyproject.toml`，其中 `[build-system]` 和 `[project]` 是新项目推荐的元数据入口。([Python Packaging][5])

### 1.2 运行模式

#### 模式 A：独立 Web 服务

```bash
obsidian-publish serve \
  --vault ~/Vaults/MyVault \
  --host 127.0.0.1 \
  --port 8008 \
  --home README \
  --reload
```

用于个人本地预览、NAS 部署、Docker 部署。

#### 模式 B：静态导出

```bash
obsidian-publish build \
  --vault ~/Vaults/MyVault \
  --out ./public \
  --base-url https://notes.example.com
```

用于 Netlify、Cloudflare Pages、GitHub Pages、Nginx 静态站点。

#### 模式 C：Django 集成

```python
# settings.py
INSTALLED_APPS += ["obsidian_publish.django_app"]

OBSIDIAN_PUBLISH = {
    "default": {
        "vault_path": "/srv/notes/vault",
        "url_prefix": "/notes/",
        "home_file": "README",
        "show_graph": True,
        "show_backlinks": True,
        "show_search": True,
    }
}
```

```python
# urls.py
from django.urls import include, path

urlpatterns = [
    path("notes/", include("obsidian_publish.django_app.urls")),
]
```

Django 的实时能力建议走 ASGI；Django 官方文档说明 async views 在 WSGI 下也能运行，但长连接、长轮询、流式响应要获得真正异步栈收益需要 ASGI。([Django Project][6]) Django Channels 可以让 Django 处理 WebSocket 等 HTTP 之外的协议，适合作为 Django 集成模式的实时更新后端。([Channels][7])

## 2. 功能矩阵

### 2.1 Perlite 功能对齐

| Perlite 能力             | 本项目实现要求                                                    | 优先级 |
| ---------------------- | ---------------------------------------------------------- | --- |
| 自动根据 vault 文件夹结构构建     | 扫描 Markdown、附件、文件夹，生成导航树                                   | P0  |
| 无数据库                   | 默认纯文件 + 内存索引；SQLite 仅作为可选缓存                                | P0  |
| Obsidian 主题支持          | 支持 `publish.css`、`.obsidian/themes/*.css` 导入、CSS variables | P2  |
| 响应式                    | 桌面三栏布局，移动端 drawer                                          | P2  |
| 不手动转换                  | 请求时渲染 + 缓存；文件变化自动失效                                        | P0  |
| 交互式 Graph              | `vis-network` 渲染全局/本地图谱                                    | P2  |
| LaTeX                  | KaTeX 或 MathJax 渲染 `$...$` / `$$...$$`                     | P1  |
| Mermaid                | Mermaid 渲染 fenced code block                               | P1  |
| Link to Obsidian Vault | 支持 `obsidian://open?vault=...&file=...` 可选按钮               | P3  |
| Search                 | 客户端 MiniSearch + 可选服务端 FTS                                 | P1  |
| Obsidian 标签/链接/图片/预览   | Wikilink、embed、tags、hover preview                          | P1  |
| Dark/Light mode        | CSS 变量 + localStorage                                      | P2  |

Perlite 的图谱依赖 Obsidian Metadata Extractor 插件生成 `metadata.json`；本项目应优先自己解析 vault 来生成 graph，同时兼容读取 `metadata.json` 作为可选加速路径。([GitHub][8]) Perlite 的 Docker 配置也体现了 `NOTES_PATH`、`HIDE_FOLDERS`、`HOME_FILE` 等核心配置，且未指定 `NOTES_PATH` 时有默认起点。([GitHub][9])

### 2.2 Obsidian Publish 功能对齐

| Obsidian Publish 能力     | 本项目实现要求                                             | 优先级 |
| ----------------------- | --------------------------------------------------- | --- |
| Site Name               | `site.name` 配置，影响 title/header/OG metadata          | P1  |
| Homepage File           | `home_file`；未配置则 README/index/Home/首个文件             | P0  |
| Logo                    | 支持 vault 内 logo 路径                                  | P2  |
| Custom Domain           | 独立部署层支持，包内只生成正确 canonical/base URL                  | P3  |
| Disallow indexing       | 生成 `robots.txt`、meta robots                         | P2  |
| Theme Light/Dark/System | `default_theme` + 用户切换                              | P2  |
| Hover Preview           | 内链 hover 弹窗，移动端点击预览                                 | P1  |
| Hide Page Title         | `hide_title`                                        | P2  |
| Readable Line Length    | CSS class toggle                                    | P2  |
| Strict Line Breaks      | Markdown 渲染配置                                       | P1  |
| Stacked Pages           | 横向 pane / sliding window，可后做                        | P4  |
| Navigation              | 文件树、排序、隐藏项                                          | P0  |
| Search Bar              | 全文搜索                                                | P1  |
| Graph View              | 右侧全局/本地图谱                                           | P2  |

Obsidian Publish 的站点选项官方列出了 Homepage File、Logo、Custom Domain、Disallow search indexing、主题、Light/Dark toggle、hover preview、hide title、readable line length、strict line breaks、stack pages、navigation、search、graph、table of contents、backlinks、passwords、Analytics 等能力。([Obsidian][10]) 其自定义能力通过 vault 根目录的 `publish.css`、`publish.js`、favicon 文件实现，并支持自定义导航顺序和隐藏导航项。([Obsidian][11])

## 3. 推荐技术栈

### 3.1 Python 后端

| 用途              | 推荐                                                        |
| --------------- | --------------------------------------------------------- |
| Python 版本       | 3.11+，最低 3.10                                             |

| Web 框架          | `starlette` 或 `fastapi` 作为独立 ASGI app                     |

| 搜索              | 默认生成 JSON 给 MiniSearch；可选 SQLite FTS5                     |
| 测试              | `pytest`, `pytest-django`, `pytest-asyncio`, `hypothesis` |
| 质量              | `ruff`, `mypy`, `pyright` optional                        |
| E2E             | `playwright`                                              |

`markdown-it-py` 是 markdown-it 的 Python port，强调可配置语法、可添加或替换规则，适合实现 Obsidian 扩展语法。([Markdown-it-py][12]) 其插件体系支持 GFM table、strikethrough、task list、alert 等，可作为 Markdown 基础层。([Markdown-it-py][13]) `watchfiles` 是现代 Python 文件监听库，可用于实时重新索引 vault。([GitHub][14])

### 3.2 前端

| 用途      | 推荐                                                 |
| ------- | -------------------------------------------------- |
| 构建      | Vite + TypeScript                                  |
| UI      | 原生 Web Components / Lit / Alpine.js，避免重型 SPA       |
| 局部更新    | htmx 或自写 fetch 模块                                  |
| 图谱      | `vis-network`                                      |
| 搜索      | `MiniSearch`                                       |
| Mermaid | `mermaid`                                          |
| 数学公式    | `KaTeX` auto-render；复杂兼容可配置 MathJax                |
| XSS 防护  | DOMPurify，用于前端动态插入 HTML 时二次净化                      |

`vis-network` 用 HTML canvas 展示节点和边，支持自定义节点、边、样式、大小和交互，并适合几千节点规模的网络图。([Vis.js][15]) Mermaid 是基于文本定义生成图表的 JavaScript 工具。([GitHub][16]) KaTeX auto-render 会扫描指定元素中的文本节点并根据 delimiter 自动渲染数学公式。([KaTeX][17]) MiniSearch 是可在浏览器和 Node 中运行的轻量全文搜索引擎。([npm][18]) DOMPurify 用于净化 HTML、MathML、SVG，防止 XSS。([GitHub][19])

## 6. 核心数据模型

### 6.1 路径与 ID 规则

每个文件都要有稳定 ID。不要直接用绝对路径作为公开 ID。

```python
def note_id(rel_path: str) -> str:
    # rel_path 使用 POSIX 风格，大小写保留
    # 例：Folder/My Note.md -> sha1("Folder/My Note.md")
```

路径规范：

1. 内部统一使用 POSIX path：`Folder/My Note.md`。
2. 禁止 `..` 路径逃逸。
3. Windows 路径进入系统后转换为 POSIX 相对路径。
4. URL 不包含 `.md` 后缀，除非附件。
5. root `/` 解析为：

   * `home_file` 配置；
   * 否则 `README.md`；
   * 否则 `index.md`；
   * 否则 `Home.md`；
   * 否则按导航排序后的第一个可见 Markdown 文件。

## 7. Obsidian Markdown 兼容策略

Obsidian Flavored Markdown 支持 CommonMark、GFM、LaTeX，并扩展了 `[[Link]]`、`![[Link]]`、block reference、footnotes、comments、strikethrough、highlight、task list、callouts、tables 等语法。([Obsidian][20]) 内部链接既支持 Wikilink，也支持 Markdown link，且可以链接 heading、block，并通过 `|` 自定义显示文本。([Obsidian][21])

### 7.1 渲染流水线

每个 Markdown 文件的处理流程：

```text
read file
  -> decode UTF-8 / UTF-8 BOM
  -> split YAML frontmatter
  -> extract properties: title, aliases, tags, publish, permalink, description, image, cover
  -> strip Obsidian comments for visible output
  -> parse structural metadata:
       headings, block ids, tags, wikilinks, markdown links, embeds
  -> resolve links against VaultIndex
  -> transform Obsidian syntax into markdown-it tokens / HTML placeholders
  -> markdown-it-py render
  -> postprocess:
       heading anchors
       callouts
       embed rendering
       Mermaid placeholders
       KaTeX placeholders
       backlinks placeholders
  -> sanitize HTML if safe_mode
  -> wrap into page template
```

### 7.2 Frontmatter / Properties

支持：

```yaml
---
title: Custom Title
aliases:
  - Old Name
  - Another Name
tags:
  - project/demo
publish: true
permalink: custom/path
description: SEO description
image: assets/card.png
cover: assets/cover.png
cssclasses:
  - wide-page
---
```

规则：

1. `title` 优先作为页面标题。
2. 没有 `title` 时，使用第一个 H1。
3. 没有 H1 时，使用文件名 stem。
4. `aliases` 用于链接解析和旧 URL redirect。
5. `tags` 与正文 inline tags 合并。
6. `publish: false` 默认隐藏该文件。
7. `publish: true` 在 `publish_true` 模式下才发布。
8. `permalink` 覆盖默认 URL，并让默认路径 301/302 跳转到 permalink。
9. `description`、`image`、`cover` 用于 SEO/OpenGraph。

Obsidian Publish 的 SEO 文档明确支持通过 Properties 配置 `publish`、`permalink`、`description`、`image`、`cover` 等页面元数据。([Obsidian][22]) Permalink 可以将默认路径重命名为自定义 URL，aliases 也可用于旧笔记路径重定向。([Obsidian][23])

### 7.3 Wikilink 语法

必须支持：

```md
[[Note]]
[[Note|显示文本]]
[[Folder/Note]]
[[Folder/Note.md]]
[[Note#Heading]]
[[Note#Heading|显示文本]]
[[Note#^block-id]]
[[#Current Page Heading]]
[[^current-page-block]]
![[Image.png]]
![[Image.png|300]]
![[Image.png|300x200]]
![[Other Note]]
![[Other Note#Heading]]
![[Other Note#^block-id]]
```

解析器要求：

1. 不解析 code fence、inline code 内的 wikilink。
2. 不解析 Obsidian comment `%% ... %%` 中的链接。
3. `![[...]]` 为 embed。
4. `[[...]]` 为普通 link。
5. `#` 后为 heading/block anchor。
6. `#^` 表示 block reference。
7. `|` 对普通链接表示 display text。
8. `|100` 或 `|100x200` 对图片 embed 表示尺寸。
9. 文件名中的 `#`、`|`、`^` 边界要遵循 Obsidian 的限制；解析失败时保留原文并给出 broken-link 样式。

### 7.4 链接解析算法

实现 `LinkResolver.resolve(source_note, target_text)`：

优先级：

1. 如果 target 是外部 URL：保持外链。
2. 如果 target 带扩展名且匹配附件：返回 attachment URL。
3. 如果 target 是相对路径：

   * 先相对当前文件夹查找；
   * 再相对 vault root 查找。
4. 如果 target 是绝对 vault 路径：相对 vault root 查找。
5. 如果 target 不带 `.md`：

   * 尝试 `target + ".md"`；
   * 尝试 basename/stem 匹配；
   * 尝试 alias 匹配。
6. 多个候选时：

   * 优先同目录；
   * 再优先最短路径；
   * 再按自然排序；
   * 同时记录 warning，供 `doctor` 输出。
7. 找不到时返回 unresolved link，HTML 中渲染为 broken link。

### 7.5 Embeds

Obsidian embed 通过在 internal link 前加 `!` 实现，图片支持 `|width` / `|widthxheight`，PDF 支持 `#page=N` 和 `#height=N` 等写法。([Obsidian][24])

实现要求：

| 类型                  | 渲染方式                                   |
| ------------------- | -------------------------------------- |
| Markdown note embed | 递归渲染目标 note 的 body，带深度限制               |
| Heading embed       | 只渲染该 heading 到下一个同级/更高级 heading 之间内容   |
| Block embed         | 只渲染 block 内容                           |
| Image               | `<img>`，支持宽高                           |
| SVG                 | 默认作为 `<img>`，不要 inline 未净化 SVG         |
| Audio               | `<audio controls>`                     |
| Video               | `<video controls>`                     |
| PDF                 | `<iframe>` 或 `<object>`，支持 page/height |
| Canvas              | 读取 `.canvas` JSON，先实现只读简化渲染            |
| Unsupported         | 渲染为下载/打开链接                             |

防递归规则：

```text
max_embed_depth = 5
if target_note in render_stack:
    render embed error: circular embed
```

### 7.6 Callouts

Obsidian callout 语法：

```md
> [!note] Title
> Content
```

支持：

```md
> [!tip] Custom title
> [!warning]+ Expanded foldable
> [!faq]- Collapsed foldable
```

Callout 支持 title、自定义 title、折叠、嵌套和默认类型；官方文档也说明 callouts 在 Obsidian Publish 中原生支持。([Obsidian][25])

实现要求：

1. 支持默认类型：note, abstract, info, todo, tip, success, question, warning, failure, danger, bug, example, quote。
2. 支持 alias：summary/tldr, hint/important, check/done, help/faq, caution/attention, fail/missing, error, cite。
3. 不认识的 callout type 降级为 note。
4. 输出结构：

```html
<div class="callout" data-callout="tip" data-callout-fold="open">
  <div class="callout-title">
    <span class="callout-icon"></span>
    <span class="callout-title-inner">Custom title</span>
  </div>
  <div class="callout-content">...</div>
</div>
```

### 7.7 Tags

支持正文标签和 frontmatter 标签：

```md
#project/demo
#会议
#inbox/to-read
```

规则：

1. 代码块、inline code、URL fragment 内不识别 tag。
2. frontmatter `tags` 必须支持 list 和 comma string。
3. tag 统一存储为小写比较、原始大小写显示。
4. nested tags 建树。
5. 点击 tag 后进入 tag 过滤页：`/tags/project/demo`。

Obsidian tags 支持正文 `#tag` 和 `tags` property，也支持用 `/` 表示 nested tags。([Obsidian][26])

### 7.8 Backlinks

实现两类：

1. `linked_mentions`：其他 note 中明确链接到当前 note。
2. `unlinked_mentions`：其他 note 正文中出现当前 note title / aliases，但没有链接；默认关闭，因为大 vault 成本高。

Obsidian backlinks 的核心定义是“从其他 note 指向当前 note 的链接”，并区分 linked mentions 与 unlinked mentions。([Obsidian][27])

### 7.9 Mermaid

支持：

````md
```mermaid
graph TD
  A --> B
```
````

后端输出：

```html
<div class="mermaid">graph TD...</div>
```

前端：

```ts
import mermaid from "mermaid";

mermaid.initialize({
  startOnLoad: false,
  securityLevel: "strict",
  theme: currentTheme === "dark" ? "dark" : "default",
});

await mermaid.run({ querySelector: ".mermaid" });
```

要求：

1. safe mode 下使用 Mermaid strict security。
2. 图表渲染失败时显示代码 fallback。
3. 文件变更或局部刷新后重新运行 Mermaid。

### 7.10 Math

支持：

```md

block $$x^2 + y^2 = z^2$$
```

前端使用 KaTeX auto-render：


## 8. 实时渲染设计

### 8.1 文件监听

使用 `watchfiles` 监听 vault：

```python
async def watch_vault(config: PublisherConfig, indexer: VaultIndexer, bus: EventBus):
    async for changes in awatch(config.vault_path):
        affected = classify_changes(changes)
        result = await indexer.apply_changes(affected)
        await bus.publish(IndexChangedEvent(result))
```

要求：

1. debounce：默认 150ms 合并多次保存事件。
2. 忽略临时文件：`.swp`, `~`, `.tmp`, `.DS_Store`。
3. 忽略 `.obsidian/workspace*` 等高频变化。
4. 删除文件时更新 backlinks、graph、search、nav。
5. 重命名通常表现为 delete + add，要能处理 redirect alias。
6. 文件变更后只重新解析受影响 note，再更新依赖该 note 的 backlinks/embeds。

### 8.2 前端实时通道

支持三档：

1. WebSocket：Django Channels / ASGI。
2. SSE：简单、适合单向更新。
3. Poll fallback：无 ASGI 时定期查询 index version。

事件格式：

```json
{
  "type": "index.changed",
  "version": 42,
  "changed": [
    {"kind": "note", "path": "README.md", "url": "/README", "change": "modified"}
  ],
  "deleted": [],
  "graph_changed": true,
  "nav_changed": false,
  "search_changed": true
}
```

前端行为：

1. 当前页面被修改：自动重新拉取 HTML partial 并替换 `.markdown-body`。
2. 当前页面被删除：显示 404，并推荐相近页面。
3. nav 变化：刷新左侧导航。
4. graph 变化：刷新 graph data。
5. search 变化：重新加载 search index。
6. backlinks 变化：刷新右侧 backlinks。
7. 不自动覆盖用户滚动位置；刷新 body 后尽量恢复 heading anchor 和 scroll offset。

## 15. 安全要求

### 15.1 路径安全

必须实现：

```python
def safe_join(vault_root: Path, rel: str) -> Path:
    candidate = (vault_root / rel).resolve()
    root = vault_root.resolve()
    if not candidate.is_relative_to(root):
        raise PathTraversalError(rel)
    return candidate
```

要求：

1. 禁止 `../` 逃逸。
2. 默认不跟随 symlink。
3. symlink 开启时也必须验证 resolved path 在允许 root 内。
4. 不暴露 vault 绝对路径。
5. 不允许直接访问 `.git`、`.obsidian`、隐藏文件、配置文件。
6. 不直接公开 `metadata.json`、原始 `.md`，除非明确配置。
7. 附件访问必须检查是否在 published set 或允许目录内。

Perlite README 也特别提醒不要让浏览器直接访问 `.md` 文件和 `metadata.json`，因为 metadata 可能包含整个 vault 结构。([GitHub][1])

### 15.2 HTML 安全

默认 `html_safe_mode = true`：

1. Markdown raw HTML 默认禁用或 sanitize。
2. `publish.js` 默认只在 trusted 模式启用。
3. Mermaid 使用 strict security。
4. SVG 不 inline，默认 `<img src>`。
5. 外链加：

```html
rel="noopener noreferrer"
target="_blank"
```

6. CSP 建议：

```http
Content-Security-Policy:
  default-src 'self';
  img-src 'self' data: https:;
  media-src 'self' https:;
  script-src 'self';
  style-src 'self' 'unsafe-inline';
  frame-src 'self';
```

如启用 `publish.js` 或 CDN，文档必须提示用户调整 CSP。

### 15.3 隐私与发布过滤

发布判断函数：

```python
def is_publishable(file: VaultFile, config: PublisherConfig) -> bool:
    if is_hidden(file) and not config.hidden_file_access:
        return False
    if matches_excluded_folder(file):
        if frontmatter_publish_true(file):
            return True
        return False
    if matches_excluded_glob(file):
        return False
    if config.publish_property_mode == "publish_true":
        return frontmatter_publish_true(file)
    if frontmatter_publish_false(file):
        return False
    return True
```

Obsidian Publish 支持 `publish: true` 自动选择发布，也支持 `publish: false` 忽略，并且 `publish: true` 可以覆盖 excluded folders；这个规则应在本项目中复刻。([Obsidian][28])

## 24. 关键实现细节

### 24.1 不要用正则直接“渲染整个 Markdown”

可以用正则提取 Obsidian 扩展语法，但完整 Markdown 渲染必须交给 Markdown parser。正确方式：

1. 用 scanner 找 fenced code / inline code ranges。
2. 在非 code ranges 内识别 wikilink、tags、comments。
3. 将 Obsidian token 转为 markdown-it token 或安全 placeholder。
4. 让 markdown-it 渲染 CommonMark/GFM。
5. 再替换 placeholder 为经过 resolver 生成的安全 HTML。

### 24.2 Cache key

```python
cache_key = (
    note.content_hash,
    config.render_relevant_hash(),
    index.version_for_links,
)
```

当以下变化发生时必须失效：

1. 文件内容变更。
2. frontmatter 变更。
3. 被 embed 的目标文件变更。
4. link resolution 结果变更。
5. config 中影响渲染的字段变更。
6. theme/css 不影响 HTML cache，但影响浏览器资源 cache。

### 24.3 Broken links

HTML：

```html
<a class="internal-link is-unresolved" data-target="Missing">Missing</a>
```

页面底部可选 debug：

```text
Broken links on this page:
- [[Missing]]
- [[Folder/Nope]]
```

### 24.4 Duplicate filenames

Obsidian 允许用 basename 链接，但 duplicate basename 会产生歧义。实现上：

1. 尽量 resolve。
2. 记录 warning。
3. `doctor` 提醒用户改为 `[[Folder/Note]]`。

### 24.5 `.obsidian` 支持

读取但不公开：

| 文件                                   | 用途                       |
| ------------------------------------ | ------------------------ |
| `.obsidian/app.json`                 | 可读取 attachment folder 设置 |
| `.obsidian/appearance.json`          | 可读取主题名                   |
| `.obsidian/themes/<theme>/theme.css` | 可选复制为 publish theme      |
| `.obsidian/snippets/*.css`           | 可选导入                     |
| `.obsidian/workspace.json`           | 忽略，不发布                   |
| `.obsidian/plugins/*`                | 默认忽略                     |

项目完成到可用状态时，必须满足：

```text
[ ] pip install 后 CLI 可用
[ ] serve 模式可打开 vault
[ ] / 默认显示 README 或首个文件
[ ] /README 可访问
[ ] 内链可点击
[ ] broken link 有样式
[ ] 图片/PDF/音频/视频附件可显示或下载
[ ] callouts 正常
[ ] Mermaid 正常
[ ] LaTeX 正常
[ ] tags 正常
[ ] backlinks 正常
[ ] TOC 正常
[ ] search 正常
[ ] graph 正常
[ ] hover preview 正常
[ ] dark/light mode 正常
[ ] 移动端可用
[ ] 修改文件后实时刷新
[ ] Django include 后可用
[ ] static export 可用
[ ] sitemap/rss/robots 可生成
[ ] publish.css 可生效
[ ] publish.js trusted mode 可生效
[ ] private/excluded 文件不可访问
[ ] path traversal 被拦截
[ ] doctor 能发现 broken links / duplicate aliases / duplicate permalinks
[ ] CI 通过 pytest/ruff/mypy
```

## 29. 最重要的实现顺序

按这个顺序做，成功率最高：

```text
1. Vault 扫描和安全路径
2. Markdown 基础渲染
3. Wikilink 解析和链接 resolver
4. 页面路由和首页规则
5. 导航树
6. Backlinks / TOC
7. 搜索
8. Embeds / Callouts
9. Graph
10. 前端布局
11. Hover preview
12. 实时 watcher
13. Django app
14. 静态导出
15. SEO / RSS / sitemap
16. 安全加固和 doctor
```

核心判断标准：先让“一个 vault 能安全、正确、快速地被浏览”成立，再逐步补齐 Perlite 和 Obsidian Publish 的高级体验。
